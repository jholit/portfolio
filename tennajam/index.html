Case Study

TennaJam – Dedicated Game-Server Hosting Application

TennaJam is a product design exploration focused on simplifying dedicated game-server setup and management. The work prioritizes clear flows, consistent structure, and user confidence over surface-level visual polish.

Server Management User Flows Wireframing Information Architecture
Approx. 8 min read
01 — Overview

1. Project Summary

TennaJam started from a simple observation: despite years of evolution in consumer applications, the experience of configuring and managing dedicated game servers still feels unnecessarily complex and fragile.

This case study focuses on the structure behind an approachable hosting experience. Instead of pushing high-fidelity visuals, the work emphasizes:

  • Predictable flow structure across the product.
  • Clear separation between overviews and detailed views.
  • Guided setup for new servers with reasonable defaults.
  • Consistent terminology for states and key actions.
02 — Problem Space

2. Problem and Goal

In many hosting panels, core information and controls are fragmented across multiple screens and tabs. Tables are dense, server states unclear, and high-impact actions poorly differentiated.

  • Users hesitate to make changes because outcomes are unclear.
  • Non-technical users struggle to complete setup without guidance.
  • Advanced settings often appear either buried or overwhelming.

The goal of TennaJam is to design a hosting and management experience that reduces cognitive load while preserving advanced control.

03 — Role & Scope

3. Role, Responsibilities, and Constraints

This was a solo case study, intended to demonstrate how a Product Designer approaches a technical problem space through structure and flows rather than visuals alone.

Responsibilities

  • Define the problem and desired outcomes.
  • Map primary user journeys and key states.
  • Design the information architecture.
  • Create low-fidelity wireframes.
  • Document decisions and trade-offs.

Infrastructure implementation details, pricing, and branding are treated as out of scope.

04 — Research

4. Research Summary

Reviewing existing hosting panels revealed recurring UX issues:

  • Dense dashboards with weak hierarchy.
  • Inconsistent naming for server states/options.
  • Critical actions distributed across tabs.
  • No single “home of truth” for a server’s status.

These findings informed TennaJam’s guiding principles:

  • Establish a central Server Management hub.
  • Represent server states consistently.
  • Expose only relevant information at each stage.
  • Keep navigation shallow.
05 — Flows

5. Core User Flows

TennaJam is structured around a focused set of narrow but deep flows addressing the most common scenarios.

Primary flow: Day-to-day management

Routine checks and interventions:

  • Home → Server Management → Server Details → Action

Users begin at an overview, identify a server, inspect its details, and act (start/stop/restart).

Secondary flow: Hosting a new server

A guided linear setup flow:

  • Select name, game, region.
  • Review recommended defaults.
  • Enable advanced settings if needed.
  • Confirm summary and launch.

Each step answers one focused question at a time.

06 — Screen Architecture

6. Wireframe-Level Overview

Key Screens

  • Home: Overview, recent activity, entry to Server Management.
  • Server Management: Sortable list of servers + state visibility.
  • Server Details: Region, uptime, stats, logs, actions.
  • Hosting Flow: Guided configuration for launching new servers.
07 — Patterns

7. Design Patterns and Decisions

Navigation & Structure

  • Keep navigation shallow.
  • Separate “scan whole system” from “act on one server”.
  • Provide one trustworthy details page per server.

States & Actions

  • Use consistent naming for states.
  • Place key actions in predictable locations.
  • Confirm all destructive/disruptive actions.

Advanced Controls

  • Group advanced options rather than scattering them.
  • Keep advanced controls available without overwhelming novices.
  • Ensure the setup flow can expand without breaking structure.
08 — Learnings

8. Key Outcomes and Learnings

Flow-first thinking leads to calmer tools with fewer reactive fixes. Aligning interface structure with user mental models makes complex actions feel simpler and safer.

Clarity and control aren’t opposites—consistent grouping, predictable actions, and well-placed confirmations allow advanced users to thrive without overwhelming beginners.
